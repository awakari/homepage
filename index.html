About

Awakari is a service that combines publish/subscribe capability with real-time search.

How it works

Awakari is capable to consume event streams from any source.

A user expresses their interest as a subscription with a certain search criteria.

For every incoming event, Awakari does the following:

  • Resolves matching subscriptions
  • Notifies the online user immediately

A user is also able to publish own events to Awakari.

    Comparison

    Event Driven

    • Push notifications reduce the latency.
    • Avoids repeated dry polls cost. ⧉

    Inverse Search

    • Finds queries by incoming event.
    • More-cost efficient for large number of queries.

    How does Inverse Search reduce costs?

    Given: K input events per second, N registered queries.

    In conventional stream data processing, every incoming event is processed by every registered query. Hence, the resulting load may be estimated as K * N. For example, AWS Kinesis Data Analytics charges per KPU ⧉ that is in direct ratio to the load.

    In case of Inverse Search, the search is executed once for every incoming event. The Inverse Search itself uses the index, hence every search operation complexity is O(log(N)). The resulting load is K * log(N).

    Quick Start

    The steps below apply to the Serverless Awakari only (coming soon).

    See the Core Usage ⧉ for an On-Premises deployment.

    Preparation Steps

    1. Install grpcurl ⧉
    2. Download the shared certificate and save to the current directory: CA file ⧉, client certificate ⧉ and client key ⧉
    3. Download the necessary proto files and save to the current directory: Cloud Events ⧉, Reader ⧉ and Writer ⧉.

    Basic Usage Walkthrough

    1. Check access: 
      grpcurl \
  -key client-shared-demo.awakari.cloud.key \
  -cert client-shared-demo.awakari.cloud.crt \
  -cacert ca-demo.awakari.cloud.crt \
  -H 'X-Awakari-User-Id: john.doe' \
  -d '{"subj": 2}' \
  demo.awakari.cloud:443 \
  awakari.api.limits.Service/Get
      A successful response looks like: 
      {
  "count": "123"
}
      The missing attribute "userId" in the response means that the limit is group-wide. A group-wide limit means it's shared with other users using the same certificate.

      Learn more about the subject options ⧉.

    2. Next, check the current group-wide usage (skipping the "X-Awakari-User-Id" header): 
      grpcurl \
  -key client-shared-demo.awakari.cloud.key \
  -cert client-shared-demo.awakari.cloud.crt \
  -cacert ca-demo.awakari.cloud.crt \
  -d '{"subj": 2}' \
  demo.awakari.cloud:443 \
  awakari.api.permits.Service/GetUsage
      A successful response looks like: 
      {
  "count": "1",
  "countTotal": "1",
  "since": "2023-06-18T14:39:18.087Z"
}
      Meaning there's 1 subscription have been spent. Hence, the remaining count of subscriptions those can be created is 123 - 1 = 122.
    3. Create a subscription: 
      grpcurl \
  -key client-shared-demo.awakari.cloud.key \
  -cert client-shared-demo.awakari.cloud.crt \
  -cacert ca-demo.awakari.cloud.crt \
  -H 'X-Awakari-User-Id: john.doe' \
  -d @ \
  demo.awakari.cloud:443 \
  awakari.subscriptions.Service/Create
      Insert the subscription input data and press ^D: 
      {
   "description": "Tesla model S updates",
   "enabled": true,
   "cond": {
      "not": false,
      "tc": {
        "key": "",
        "term": "Tesla Model S",
        "exact": false
      }
    }
}
      A successful response looks like: 
      {
  "id": "547857e3-adfc-48a5-a49e-110cfdedbaab"
}
      Note the created subscription id and use it further to read the messages.

      Learn more about the Subscriptions API ⧉.

    4. Start reader stream: 
      grpcurl \
  -proto reader.proto \
  -max-time 86400 \
  -key client-shared-demo.awakari.cloud.key \
  -cert client-shared-demo.awakari.cloud.crt \
  -cacert ca-demo.awakari.cloud.crt \
  -H 'X-Awakari-User-Id: john.doe' \
  -d @ \
  demo.awakari.cloud:443 \
  awakari.reader.Service/Read
      Specify the subscription id in the request and press Enter: 
      {"start": {"batchSize": 1, "subId": "547857e3-adfc-48a5-a49e-110cfdedbaab"}}
    5. Switch to another shell/window and start the writer stream: 
      grpcurl \
  -proto writer.proto \
  -key client-shared-demo.awakari.cloud.key \
  -cert client-shared-demo.awakari.cloud.crt \
  -cacert ca-demo.awakari.cloud.crt \
  -H 'X-Awakari-User-Id: john.doe' \
  -d @ \
  demo.awakari.cloud:443 \
  awakari.writer.Service/SubmitMessages
      Then publish an event: 
      {
   "msgs": [
      {
         "id": "3426d090-1b8a-4a09-ac9c-41f2de24d5ac",
         "type": "example.type",
         "source": "example/uri",
         "spec_version": "1.0",
         "attributes": {
            "subject": {
               "ce_string": "Tesla price updates"
            },
            "time": {
               "ce_timestamp": "2023-07-03T23:20:50.52Z"
            }
         },
         "text_data": "Tesla model S is now available at lower price"
      }
   ]
}
      A successful response looks like: 
      {
  "ackCount": 1
}
      After this, you can close the writer stream by pressing ^C or leave it open to publish any other messages later.
    6. Switch back to the reader stream window/shell and observe the message is received
    7. Acknowledge the received message by typing: 
      {"ack": { "count": 1}}
      After this, you can close the reader stream by pressing ^C or leave it open to read any other messages later.
    8. That's it! Delete the subscription to clean up: 
      grpcurl \
  -key client-shared-demo.awakari.cloud.key \
  -cert client-shared-demo.awakari.cloud.crt \
  -cacert ca-demo.awakari.cloud.crt \
  -H 'X-Awakari-User-Id: john.doe' \
  -d '{"id": "547857e3-adfc-48a5-a49e-110cfdedbaab"}'
  demo.awakari.cloud:443 \
  awakari.subscriptions.Service/Delete

    Get Access

    Prepare a certificate request:

    openssl req -new -newkey rsa:4096 -nodes \
  -keyout client.key \
  -out client.csr \
  -addext "subjectAltName=DNS:subscriptions.awakari.cloud" \
  -subj '/CN=group0.company1.com'

    Note: Never specify additional certificate attributes like "O", "OU", etc. The resulting DN should not contain commas.

    Send the certificate request by email ⧉. The requested certificate will be in the response email.
